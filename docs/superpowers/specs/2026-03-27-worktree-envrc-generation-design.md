# Worktree `.envrc` Generation

## Problem

When a user manually enters a flowyeah worktree directory (e.g., to run the app for manual testing), the worktree-isolated environment variables configured in `flowyeah.yml` are not available in their shell. This causes the app to use the main checkout's development database, leading to schema corruption when different branches run different migrations.

The env vars are resolved and persisted in the session state file (`state.md`), but only Claude's session exports them. The user's shell has no way to pick them up.

## Solution

During the worktree lifecycle Environment step, after resolving env vars and persisting them to the state file, also write a `.envrc` file in the worktree root.

### `.envrc` format

```bash
# Generated by flowyeah - do not edit
export DB_SUFFIX=kM4tQ8hN
export REDIS_DB=pL7nR2wY
```

### direnv integration

If `direnv` is available on `PATH`, run `direnv allow .` after writing the file. This activates automatic env loading when the user `cd`s into the worktree.

Users without direnv can `source .envrc` manually.

### When to generate

Only when `worktree.env` in `flowyeah.yml` has entries. If `worktree.env` is empty or absent, no `.envrc` is written.

### Teardown

No changes needed. The `.envrc` is deleted when the worktree directory is removed.

## Scope

- **Changed file**: `worktree-lifecycle.md` (Environment step)
- **No new config keys**
- **No schema changes**
- **No new dependencies**
